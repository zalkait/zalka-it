ZalkaIT — migracja kostki OLAP → SSAS Tabular

Plan współpracy z Claude

Narzędzia, podział ról i zasady pracy na czas projektu — od projektu modelu po wdrożenie ETL i raporty.

Krótki plan

  1. Ty instalujesz: SSMS, SSDT (workloady Integration Services + Analysis Services), Excel z Power Query — to obsłuży ETL, kostkę i raporty w jednym środowisku.
  2. Ja projektuję: model DWH, model Tabular, katalog miar DAX, architekturę ETL i piszę kod (T-SQL, DAX, Power Query M) na bazie tego, co mi dostarczysz.
  3. Ty budujesz i testujesz: w swoim środowisku, na realnych danych — nie mam dostępu do sieci klienta ani do bazy Technique.
  4. Pracujemy iteracyjnie: kod od mnie → Ty testujesz → wracasz z wynikiem albo błędem → poprawiam.
  5. Plan Pro wystarczy do większości tej pracy, łącznie z dodatkiem Claude for Excel.
  6. Nie zbuduję prawdziwego .dacpac — to wymaga Twojego SSDT/MSBuild. Przygotuję pliki źródłowe, które Ty zbudujesz lokalnie.

Materiały do przekazania

01Narzędzia po Twojej stronie

Podstawa — obowiązkowe

SSMS SSDT (Integration Services) SSDT (Analysis Services) Excel + Power Query

SSDT w Visual Studio z dwoma obciążeniami — Integration Services (pakiety SSIS) i Analysis Services (model Tabular) — pokryje całą warstwę budowy: ETL i kostkę w jednym środowisku.

Zalecane — darmowe, bardzo ułatwiają pracę

Tabular Editor DAX Studio Azure Data Studio

Tabular Editor (wersja 2 jest darmowa) edytuje model Tabular poza Visual Studio — przy katalogu 70+ miar DAX to oszczędność realnych godzin względem natywnego edytora SSDT. DAX Studio służy do testowania i profilowania zapytań DAX — pokazuje plan zapytania, świetne do debugowania miar, które zwracają nie to, co powinny.

Do automatyzacji i wersjonowania — opcjonalne

Git SqlPackage.exe

Git do trzymania projektu SSDT (.sqlproj, .smproj) pod kontrolą wersji — przyda się szczególnie, jeśli zechcesz, żebym pracował na plikach poprzez Claude Code. SqlPackage.exe (albo dotnet build z pakietem Microsoft.Build.Sql) do budowania i wdrażania dacpac z linii komend.

02Narzędzia Claude — co zainstalować

Claude.ai (ten czat)

Tu robimy projektowanie, kod i dokumentację. Wersja desktopowa jest wygodniejsza przy pracy z plikami (upload/download), ale funkcjonalnie web, desktop i mobile to to samo.

Claude for Excel dodatek

Instalowany z Microsoft Marketplace jako add-in do Excela, dostępny na planie Pro od stycznia 2026. Działa w panelu bocznym, czyta i edytuje już otwarty skoroszyt — pivot tables, wykresy, formuły, formatowanie warunkowe. Nie otwiera i nie tworzy plików samodzielnie. Dobrze nada się do polerowania wzorcowych raportów, które razem zaprojektujemy.

Claude Code terminal

Jeśli zechcesz, żebym pracował bezpośrednio na plikach Twojego projektu na dysku (cały folder .sql, .dtsx, .bim/.tmdl pod Git) zamiast kopiować kod w czacie tam i z powrotem. Wymaga planu Pro lub wyższego. Największy skok wygody przy większym repozytorium — nieobowiązkowy na start.

03W czym pomogę, a w czym nie

Pomogę

  • Projekt modelu DWH — schemat gwiazdy, fakty, wymiary, relacje
  • Projekt modelu Tabular i katalog miar DAX
  • Architektura ETL — wzorzec staging → SP → DWH → odświeżenie SSAS → job
  • Pisanie i poprawianie kodu: T-SQL, DAX, Power Query M
  • Debugowanie — wklej błąd, plan zapytania, próbkę danych
  • Dokumentacja, specyfikacje, diagramy procesów

Nie mogę

  • Połączyć się z serwerem klienta czy bazą Technique — brak dostępu do sieci i sterowników
  • Zbudować skompilowanego .dacpac — wymaga Twojego SSDT/MSBuild
  • Przetestować kod na żywych danych klienta — testy zawsze po Twojej stronie
  • Zbudować pełny pakiet .dtsx graficznie — mogę zaprojektować logikę albo napisać kod Biml, finalny pakiet budujesz w SSDT

04Jaki kod razem zrobimy

  • T-SQL — DDL tabel wymiarów/faktów, procedury składowane (mamy już szkielety: usp_SplitLaborByMachine, usp_LoadFactPressOperations), widoki izolujące ETL od zmian w Technique
  • DAX — katalog miar z przesłanej specyfikacji (100+ pól), logika Time Intelligence, relacje wiele-do-wielu
  • Power Query M — transformacje danych w Excelu, jeśli wolisz część logiki tam niż w kostce
  • Biml (opcjonalnie, z BimlExpress) — generowanie wielu podobnych pakietów SSIS z jednego szablonu
  • Skrypty pomocnicze — PowerShell do automatyzacji deployu, polecenia SqlPackage do budowy dacpac

05Czego potrzebuję od Ciebie

  • Schemat źródłowy z Technique — najlepiej skrypt CREATE TABLE (SSMS → prawy klik na bazę → Tasks → Generate Scripts) albo lista tabel/kolumn/typów
  • Próbki danych — kilka–kilkanaście wierszy per tabela, zanonimizowane jeśli trzeba (np. bez nazwisk operatorów)
  • Pełny tekst błędu, gdy coś nie działa — dokładny komunikat z SSMS/SSDT, zrzut ekranu też wystarczy, mogę czytać obrazy
  • Aktualny stan modelu, jeśli już coś zbudowałeś — plik .bim/.tmdl albo opis tego, co istnieje
  • Plik Excela ze specyfikacją, który już mamy — to wciąż główna podstawa katalogu miar i wymiarów

06Czy plan Pro wystarczy

Tak, do większości tej pracy. Plan Pro daje dostęp do Claude for Excel (dostępny dla wszystkich użytkowników Pro od stycznia 2026) oraz do Claude Code. Limit sesyjny resetuje się co 5 godzin — przy względnie krótkich wiadomościach można liczyć na około 45 wiadomości w takim oknie; długie wklejane schematy, duże pliki i intensywne sesje z Claude Code zużywają limit szybciej.

Jeśli kiedyś zaczniesz pracować całymi dniami w trybie agentowym (Claude Code), warto rozważyć plan Max. Na start — do projektowania modelu i pisania kodu w tym czacie — Pro powinien wystarczyć. Aktualne liczby zawsze sprawdzisz w Ustawienia → Użycie na swoim koncie, bo Anthropic koryguje je od czasu do czasu.

07Jak pracować ze mną optymalnie

  • Osobne wątki per temat — jeden do ETL/T-SQL, drugi do modelu Tabular/DAX, trzeci do Excela. Długie, rozrastające się wątki kosztują więcej tokenów na każdą kolejną wiadomość, bo cały kontekst jest przeliczany na nowo
  • Pliki zamiast wklejania — wgrywaj pliki (jak z Excelem) zamiast wklejać ogromne bloki tekstu w wiadomości
  • Pamięć między sesjami — pamiętam ustalenia z poprzednich rozmów, nie musisz powtarzać kontekstu projektu od zera w nowym wątku
  • Konkretne pytania — "napisz SP do X z logiką Y" działa lepiej i taniej niż "zrób mi cały ETL"
  • Claude Code do dużego kodu — jeśli projekt SQL/SSIS urośnie, praca na realnym repozytorium jest efektywniejsza niż kopiowanie plików w czacie
